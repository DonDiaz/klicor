# Diseño de páginas para productos cloud

## Objetivo

Definir cómo debe verse y estructurarse una página principal para un producto SaaS o cloud, con foco en:

- claridad del producto
- confianza operativa
- lectura rápida
- conversión hacia registro o demo
- consistencia entre narrativa comercial y realidad del producto

## Principios base

### 1. El producto debe verse arriba, no solo describirse

En un producto cloud, la página principal no puede depender solo de promesas. Debe enseñar temprano:

- qué hace el producto
- cómo se usa
- qué parte del trabajo simplifica

Para Klicor esto significa mostrar la página pública, el QR, el panel y la lógica de operación, no solo hablar de enlaces.

### 2. La jerarquía visual manda la lectura

La tipografía, el peso, la escala y el espacio deben guiar el orden de lectura. Si todo compite, nada se entiende rápido.

Para Klicor:

- una promesa principal fuerte
- una explicación secundaria breve
- módulos de producto bien separados
- CTAs visibles, pero no repetidos sin control

### 3. Consistencia no es repetición plana

Una página de producto debe repetir patrones para dar previsibilidad, pero permitir bloques distintos según la función:

- hero
- arquitectura del producto
- flujo de uso
- casos de uso
- precio
- CTA final

Cada bloque debe sentirse parte del mismo sistema, no una sucesión de tarjetas marketing.

### 4. Menos decoración, más evidencia

Las imágenes deben aportar comprensión. En móvil especialmente, un visual decorativo que no explica nada solo hace más lenta la página y estira el scroll.

Para Klicor:

- el preview del producto sí aporta valor
- fondos, gradientes y brillo solo deben acompañar
- el diseño debe priorizar estructura antes que adorno

### 5. Un producto cloud transmite confianza por estructura

La confianza no sale solo del logo o del color. Sale de que la página muestre:

- orden
- estabilidad
- control
- claridad funcional

Eso se logra con:

- módulos limpios
- títulos concretos
- bloques que expliquen operación real
- menos lenguaje ambiguo de marketing

### 6. El responsive debe ser por componentes

No basta con encoger una landing. Los bloques deben reorganizarse con sentido según su ancho y contexto.

Para Klicor:

- hero adaptable
- tarjetas que colapsan sin romper jerarquía
- CTAs claros en móvil
- previews que sigan siendo informativos

### 7. La página debe vender la categoría correcta

Si Klicor no quiere verse como link-in-bio genérico, el home no debe parecer una landing genérica.

Debe comunicar algo más cercano a:

- presencia operativa
- página pública para negocio
- contacto, cobro y distribución en un solo sistema

No solo todos tus links en un lugar.

## Estructura recomendada para un home cloud

### Hero

Debe responder rápido:

- qué es
- para quién es
- qué resuelve
- cuál es la acción principal

### Banda de valor inmediato

Un bloque corto que deje claro el beneficio estructural:

- una presencia pública clara
- una forma simple de compartir
- una manera ordenada de convertir

### Arquitectura del producto

Sección donde el producto se explica como sistema:

- página pública
- QR
- dashboard
- contacto y pago

### Flujo operativo

No solo cómo funciona, sino qué hace el usuario en la práctica y cómo evoluciona el uso del producto.

### Casos de uso

Sirve para que el visitante se ubique rápido:

- emprendedor
- negocio físico
- marca personal

### Precio

Debe ser simple y no pelear con la comprensión del producto.

### CTA final

No debe repetir exactamente el hero; debe cerrar con una versión más madura del beneficio.

## Criterios de calidad para Klicor

Una buena página principal para Klicor debería:

- parecer producto, no plantilla
- mostrar estructura y operación real
- permitir entender el producto en menos de 20 segundos
- sentirse consistente con dashboard y login
- usar el preview como evidencia, no como decoración
- aguantar bien móvil y desktop

## Traducción práctica al proyecto

### Lo que debe comunicar el home nuevo

- Klicor es una presencia pública operativa para negocios
- sirve para compartir, contactar y cobrar mejor
- tiene una base técnica estable
- es simple de usar, pero no superficial

### Lo que debe evitar

- exceso de bloques iguales
- promesas vagas
- demasiada sensación de landing de curso o plantilla
- hero bonito pero poco informativo
- imágenes que no expliquen el producto
